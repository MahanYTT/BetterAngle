SECTION 1 — WHAT IS THE GHOST WALK BUG
================================================================

In plain English:

  The player holds W to run. A FOV transition occurs (skydiving
  to gliding, or gliding to skydiving). BetterAngle fires
  BlockInput(TRUE) to freeze keyboard input for 700ms or 1000ms
  so that mouse twitches do not corrupt the angle calculation
  during the transition.

  If the player releases W during that freeze window, the
  release event is destroyed by Windows. Fortnite never receives
  the "key up" signal. When the freeze ends, the character keeps
  running forward forever even though no key is physically held.

  This is called the Ghost Walk.

Why BlockInput exists:
  BetterAngle tracks the mouse angle using Raw Input. During a
  FOV change (skydive <-> glide), the sensitivity must be
  recalculated. If the player moves the mouse during this
  recalculation, the angle reading is corrupted. BlockInput
  prevents that. The 700ms and 1000ms durations are the minimum
  tested values for the FOV interpolation to complete in
  Fortnite. They cannot be reduced without breaking the angle
  calculation. This was tested and confirmed.


================================================================
SECTION 2 — WHY CLOSING THE APP SEEMS TO FIX IT
================================================================

When the player closes BetterAngle between matches, the ghost
walk does not occur on the next match. This seemed like evidence
that closing the app "flushes" something.

THE ACTUAL REASON:
  When the app is closed, BlockInput is never called again.
  The ghost walk requires BlockInput(TRUE) to exist. Without the
  app running, there is no freeze, so there is no swallowed key
  release, so there is no ghost. The app closing does not fix a
  corrupted state. It removes the cause of the corruption.

  This distinction matters because it means there is no magic
  "flush" that happens on app death that we can replicate. The
  kernel destroys the swallowed events — it does not buffer them
  for later delivery.

IMPORTANT IMPLICATION:
  Every solution that tries to recover the swallowed key release
  AFTER BlockInput(FALSE) is working with incomplete information.
  The event is gone. The only ground truth available is:
    - preState: key states captured BEFORE BlockInput(TRUE)
    - GetAsyncKeyState: read AFTER the OS table thaws post-unlock
  The delta between these two is the best approximation possible.


================================================================
SECTION 3 — WHY BLOCKINPUT(FALSE) DOES NOT FIX IT AUTOMATICALLY
================================================================

Logical question: if BlockInput(TRUE) causes the freeze and
BlockInput(FALSE) ends it, why does the ghost persist?

  BlockInput(FALSE) opens the pipeline going FORWARD.
  It does not replay events that were dropped during the freeze.
  The W key release happened 400-900ms ago during the lock.
  That event is gone. Opening the pipeline again does not
  retrieve it.

  Analogy: a dam closes a river. Water that went over the
  spillway while the dam was shut is gone. Opening the dam
  again lets new water through but does not bring back what
  was lost.


================================================================
SECTION 4 — WHY RAW INPUT IS NOT THE SOLUTION DURING THE LOCK
================================================================

The theory was: use Raw Input (WM_INPUT / hardware-level events)
as a "source of truth" that bypasses the frozen OS table.

CONFIRMED FINDING (Finding #5 from audit):
  BlockInput(TRUE) also blocks WM_INPUT messages. MsgWndProc
  receives zero keyboard events during the lock. Both
  g_rawKeyUpDetected and g_rawKeyMakeDetected will be false
  for the entire lock duration regardless of what the player
  does physically.

  Raw Input is blind during the lock. It cannot serve as a
  real-time source of truth while BlockInput is active.

WHAT RAW INPUT CAN STILL DO:
  After BlockInput(FALSE), any buffered WM_INPUT events that
  were queued during the lock may arrive as a burst. The v6
  forensic overlay (RAW DATA row) is designed to detect this.
  If a burst arrives, it means events were buffered not dropped
  — and the timing of that burst tells you how long to wait
  before trusting the data.


================================================================
SECTION 5 — THE "FAKE DEATH" THEORY AND ITS LIMITS
================================================================

THEORY:
  Destroying g_hMsgWnd and unregistering Raw Input devices
  might trick Windows into flushing its input buffer the same
  way it does when a process terminates. This would deliver
  the swallowed W release to Fortnite automatically.

WHY IT MIGHT WORK:
  Windows performs kernel-level cleanup when a window is
  destroyed. Part of that cleanup may include flushing pending
  input associated with that window's registration.

WHY IT MIGHT NOT WORK:
  There is no Win32 documentation guaranteeing this behavior.
  Windows may only flush input buffers on full process
  termination via kernel exit, not on DestroyWindow. This is
  an experiment, not a known fix.

CRITICAL THREAD SAFETY ISSUE (already fixed in the plan):
  DestroyWindow MUST be called from the thread that created
  the window. The lock runs in a detached std::thread. Calling
  DestroyWindow from that thread causes undefined behavior or
  a crash. The solution is to use SendMessage/PostMessage to
  delegate window destruction to the main thread (HUDWndProc).

THE HONEST ASSESSMENT:
  The fake death is worth testing. But it is an experiment
  with a binary outcome. If it fails, the pivot is clear.
  It is not bulletproof because bulletproof requires
  recovering events that the Windows kernel destroyed.
  That API does not exist.

SECTION 6 — COMPLETE LIST OF ALL BUGS FOUND AND FIXED
================================================================

These fixes are already applied in the FINAL codebase
(BetterAngle-FINAL.zip). They are listed here for reference.

--- Input.cpp ---

BUG 1: Out-of-bounds array access (UB)
  g_gamingKeys has 4 elements (W/A/S/D). Two for loops used
  i < 5, reading index 4 which is out of bounds. Undefined
  behavior that silently corrupts the stack.
  FIX: Both loops changed to i < 4 / std::size(g_gamingKeys).

BUG 2: Unreachable keybd_event log
  g_activeFallback was set to 99, then a separate block checked
  if (g_activeFallback == 3). 99 != 3, so that block could
  never execute. The keybd_event escalation happened but was
  never logged.
  FIX: The ==3 log block moved inside the anyStillHeld branch
  after g_activeFallback = 3 is set. Now reachable.

BUG 3: FB4 early return skips g_hasSynced
  The FB4 (50ms thaw wait) path returned early on success.
  g_hasSynced = true sits at the bottom of the function.
  Result: overlay permanently showed "AWAITING FIRST LOCK"
  after every successful FB4 resolution.
  FIX: g_hasSynced = true added before the FB4 return.

BUG 4: FB4 early return skips RegisterRawMouse
  Same early return issue. RegisterRawMouse(g_hMsgWnd) at the
  bottom of the function was skipped, breaking Raw Input mouse
  tracking after every FB4 cycle.
  FIX: RegisterRawMouse called before the FB4 return.

BUG 5: Dead stateAfterWait mutation
  After FB4 injection, code wrote stateAfterWait[i] = false
  "to reflect injection" — but stateAfterWait is a local
  vector and the function returns immediately after. Nothing
  reads it. Dead code.
  FIX: Removed.

BUG 6: Dead tableThawed variable
  bool tableThawed = true was declared, never written after
  init, never read. Pure dead code.
  FIX: Removed.

BUG 7: Unconditional Sleep(50) on every lock
  The 50ms wait fired on every FOV transition even when no
  gaming key was held. On a 700ms lock this adds 50ms of
  unnecessary delay even when the user is not pressing WASD.
  FIX: Guarded by anyKeyWasHeld check. Sleep(50) only fires
  if at least one of W/A/S/D was held before the lock.

--- State.cpp ---

BUG 8: g_hudX / g_hudY defaulting to 40
  Hardcoded default of 40 placed the HUD and debug panel at
  the far left edge of the screen on first launch. The green
  GLIDING/DIVING status box also appeared on the left.
  FIX: Global defaults changed to -1 (sentinel). LoadSettings
  uses -1 as the "missing" default. WinMain detects sentinel
  and centers the HUD on first launch using SM_CXSCREEN.

BUG 9: LoadSettings eInt default of 40 for hudX/hudY
  Even with sentinel globals, LoadSettings would overwrite
  them with 40 if the key was missing from settings.json.
  FIX: eInt("hudX", -1) / eInt("hudY", -1). Only applies
  the loaded value if >= 0.

--- BetterAngle.cpp ---

BUG 10: Alt-tab path called BlockInput(TRUE/FALSE)
  The FocusMonitorThread detected alt-tab return and spawned
  a thread that called BlockInput(TRUE) -> Sleep(400) ->
  BlockInput(FALSE) with NO SyncGamingKeysNitro after it.
  This froze the key table for 400ms on every alt-tab with
  no cleanup — guaranteed ghost input on every alt-tab.
  FIX: Entire detached thread and BlockInput calls removed.
  Replaced with plain g_mouseSuspendedUntil cooldown only.
  Windows restores key state automatically on focus return.

BUG 11: Both raw key arrays not reset before lock
  g_rawKeyMakeDetected was not zeroed before each lock,
  showing stale data from previous lock cycles in the overlay.
  FIX: Both g_rawKeyUpDetected and g_rawKeyMakeDetected reset
  to false in both FOV transition threads before BlockInput.

--- Overlay.cpp ---

BUG 12: HUD renders at negative coordinates
  With sentinel -1 as default for g_hudX/g_hudY, DrawOverlay
  could render the HUD off-screen before WinMain sets the
  centered position.
  FIX: Defensive clamp added:
    int rx = (g_hudX >= 0) ? g_hudX : 40;
    int ry = (g_hudY >= 0) ? g_hudY : 40;

--- Overlay_114.cpp / BetterAngle_114.cpp ---
  These files are NOT compiled. CMakeLists.txt builds
  src/main_app/BetterAngle.cpp and src/shared/Overlay.cpp.
  The _114 files are dead code and were left untouched.

SECTION 7 — THE V6 FORENSIC OVERLAY (ALREADY IMPLEMENTED)
================================================================

Two new rows added to the debug overlay (column 1, rows 16/17):

ROW: RAW DATA
  Format: W:[MB] A:[..] S:[..] D:[..]
  M = g_rawKeyMakeDetected is true (hardware press/typematic)
  B = g_rawKeyUpDetected is true (hardware release)
  . = no event captured this cycle
  Color: turns RED if any key shows [..] after a lock —
         that is the silence case (events dropped, not buffered)

ROW: OS ASYNC
  Format: W:[v] A:[.] S:[.] D:[.]
  v = GetAsyncKeyState reports key as DOWN
  . = GetAsyncKeyState reports key as UP
  Color: turns YELLOW if OS reports key as down while
         g_physicalKeys polling says it is up — frozen table

HOW TO READ THE RESULTS:

  RAW [..] + OS [v]  → Silence case: buffer empty, OS frozen.
                        Ghost walk will occur. Neither source
                        can be trusted.

  RAW [.B] + OS [.]  → Clean release: hardware event captured,
                        OS table agrees. Existing fallback chain
                        should catch this correctly.

  RAW [M.] burst after unlock → Events were buffered during
                        lock and delivered on BlockInput(FALSE).
                        Timing of burst tells you thaw latency.

  RAW [MB] + OS [.]  → Full tap captured (press + release both
                        survived). User released during lock but
                        the break event arrived post-unlock.

SECTION 8 — THE THREE DIAGNOSTIC TESTS TO RUN
================================================================

Run these in a real Fortnite match with the debug overlay open.

TEST A — The Buffered Burst
  Action: Hold W, trigger a glide transition, KEEP holding W.
  Watch: Does RAW DATA show [..] during the lock and then
         suddenly show [M.] the frame the lock ends?
  YES → Events are buffered during lock, delivered on unlock.
        The fake death may flush them.
  NO  → Events are truly dropped. Fake death will not help.

TEST B — The Silent Release
  Action: Hold W, trigger a glide transition, release W
          halfway through the lock.
  Watch: Does RAW DATA ever show [.B]? Does OS ASYNC flip
         from [v] to [.] after the lock ends?
  RAW [.B] seen → Break event survived somehow. OS table
                  may catch it. FB4 thaw wait may be enough.
  RAW [..] + OS catches up → Events dropped but OS self-heals.
                  Measure how long OS ASYNC takes to flip.
  RAW [..] + OS stays [v] → Full silence case. Worst scenario.
                  Ghost walk guaranteed without injection.

TEST C — The Thaw Speed
  Action: Hold W, trigger transition, release W, watch landing.
  Watch: How many frames does OS ASYNC stay [v] after unlock
         before flipping to [.]?
  <10ms  → FB4 50ms wait is way too long. Reduce to 15ms.
  ~50ms  → FB4 is correctly tuned. Keep as is.
  >50ms  → FB4 fires too early. OS table still frozen when
            we read it. Increase wait or run full FB1/FB2
            chain before reading.

SECTION 9 — THE FAKE DEATH IMPLEMENTATION (NEXT BUILD)
================================================================

This is the next thing to build and test. It is not in the
FINAL zip yet. Add it to the FINAL codebase.

WHAT IT DOES:
  After BlockInput(FALSE), signal the main thread to destroy
  and recreate g_hMsgWnd. This forces Windows to perform
  OS-level cleanup on the input connection, which may flush
  any buffered input events into Fortnite's queue.

  It is an experiment. If it works, the ghost walk is solved.
  If it fails, the pivot is the unconditional KeyUp safety net
  which is included in the same code path.

--- CHANGE 1: In both FOV transition threads (BetterAngle.cpp)
    After BlockInput(FALSE) and before SyncGamingKeysNitro: ---

    // Signal main thread to perform Fake Death flush.
    // SendMessage (not PostMessage) guarantees the main thread
    // executes the flush BEFORE SyncGamingKeysNitro reads state.
    SendMessage(g_hHUD, WM_USER + 42, 0, 0);

    NOTE: Use SendMessage not PostMessage. PostMessage is async
    and the flush may not complete before SyncGamingKeysNitro
    runs, creating a race condition.

--- CHANGE 2: In HUDWndProc switch(message) (BetterAngle.cpp) ---

    case WM_USER + 42:
    {
        // SAFETY NET: Unconditional KeyUp for all WASD keys.
        // This fires regardless of whether the fake death works.
        // It is the guaranteed layer underneath the experiment.
        static const int keys[] = {'W', 'A', 'S', 'D'};
        for (int vk : keys) {
            INPUT input = {0};
            input.type = INPUT_KEYBOARD;
            input.ki.wVk = (WORD)vk;
            input.ki.wScan = (WORD)MapVirtualKeyW(vk, MAPVK_VK_TO_VSC);
            input.ki.dwFlags = KEYEVENTF_SCANCODE | KEYEVENTF_KEYUP;
            SendInput(1, &input, sizeof(INPUT));
        }

        // FAKE DEATH: Destroy and recreate the input listener.
        // This forces OS-level cleanup that may flush the buffer.
        // No Sleep() here — sleeping on the main thread freezes
        // the UI, message pump, overlay painting, and timers.
        if (g_hMsgWnd) {
            DestroyWindow(g_hMsgWnd);
            g_hMsgWnd = NULL;
        }

        // RESURRECTION: Rebuild immediately.
        // GetWindowLongPtr retrieves hInstance without needing
        // it in scope (HUDWndProc does not have it directly).
        HINSTANCE hInst =
            (HINSTANCE)GetWindowLongPtr(hWnd, GWLP_HINSTANCE);
        g_hMsgWnd = CreateWindowEx(
            0, L"BetterAngleMsgWnd", NULL, 0,
            0, 0, 0, 0, HWND_MESSAGE, NULL, hInst, NULL);

        // RE-HOOK: RegisterRawMouse registers both mouse and
        // keyboard. It is already in the codebase. Use it.
        RegisterRawMouse(g_hMsgWnd);

        return 0;
    }

WHERE IN THE FILE:
  Both changes go in src/main_app/BetterAngle.cpp only.
  HUDWndProc is at approximately line 395.
  The FOV transition threads are at approximately lines 130
  and 165. Add the SendMessage line in both threads.

IMPORTANT: Do NOT add Sleep() anywhere in this code path.
  - No Sleep in the lock thread after SendMessage (SendMessage
    already blocks until main thread processes it).
  - No Sleep in the WM_USER+42 handler (main thread freeze).

SECTION 10 — HOW TO READ THE FAKE DEATH TEST RESULTS
================================================================

Run the exact sequence in Fortnite:
  1. Hold W to run forward.
  2. Trigger a glide transition (jump off a height).
  3. While in the air and frozen, physically release W.
  4. Let the lock expire naturally and watch the character land.

RESULT: CHARACTER STOPS MOVING
  The fake death worked. Destroying g_hMsgWnd triggered a
  Windows buffer flush that delivered the W release to Fortnite.
  Next step: test reliability across 20+ landings. If consistent,
  ship it. The ghost walk is dead.

RESULT: CHARACTER KEEPS WALKING
  Windows did not flush the buffer on window destruction. It
  requires full process termination. The fake death route is
  closed. The unconditional KeyUp safety net (already in the
  WM_USER+42 handler above) still fired, so check if THAT
  stopped the ghost. If yes, the safety net alone is sufficient.
  If no, proceed to Section 11.


================================================================
SECTION 11 — FALLBACK PLAN IF EVERYTHING ABOVE FAILS
================================================================

If the fake death fails AND the unconditional KeyUp still
results in ghost walking, there are two remaining options:

OPTION A — Accept the double-press tradeoff
  Send unconditional KeyUp for all WASD on every unlock.
  If the player is still holding W, they experience a brief
  stutter and must re-press. This is the "known bad" UX that
  was previously rejected. But it is 100% ghost-free.
  Consider adding it as a user-toggleable option.

OPTION B — Remove BlockInput from the keyboard pipeline
  Use g_mouseSuspendedUntil to suppress the angle calculation
  for the full 700ms/1000ms window. Do NOT call BlockInput at
  all. The keyboard pipeline is never frozen. All key releases
  reach Fortnite instantly. Ghost walk is physically impossible.
  The tradeoff: mouse twitches during the FOV transition may
  slightly corrupt the angle reading. You need to measure in
  practice whether this is noticeable in a real match.
  Given that the transition only lasts 700-1000ms and the
  player is typically focused on landing, this may be
  acceptable. Test it.

CRITICAL RULES — NEVER BREAK THESE
================================================================

1. BlockInput(TRUE) and BlockInput(FALSE) MUST be called from
   the SAME thread. Current code uses a mutex and detached
   threads correctly. Do not change this.

2. DestroyWindow MUST be called from the thread that created
   the window (main thread). Never call it from the lock thread.
   Always use SendMessage/PostMessage to delegate to HUDWndProc.

3. Never Sleep() inside HUDWndProc or any main thread message
   handler. It freezes the entire UI and message pump.

4. VK_SPACE must never be in the gaming keys array. Space
   triggers FOV transitions. Including it in the sync loop
   causes a feedback loop (confirmed: 39 rapid locks in testing).

5. g_gamingKeys contains exactly 4 keys: W, A, S, D.
   Any loop over this array must use std::size(g_gamingKeys)
   or i < 4. Never hardcode i < 5.

6. RegisterRawMouse(g_hMsgWnd) must be called after any
   window destruction or recreation. It registers both mouse
   and keyboard raw input. Never skip this step.

7. Pre-snapshot (GetGamingKeyState) MUST happen before
   BlockInput(TRUE). It is the only clean read. Everything
   after BlockInput(TRUE) is frozen or unreliable.

8. The 500ms cooldown guard on lock re-entry must stay.
   It prevents feedback loops if anything accidentally
   triggers a second FOV transition during a sync cycle.
